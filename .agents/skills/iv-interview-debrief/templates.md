# 输出模板

## 问答 txt 模板

```markdown
# 面试问答整理
文件: 新录音 4_剪辑1.m4a

---

## 自我介绍

**问：** 方便做一下自我介绍吗？

**答：** …

---

## NSQ 消息队列

**问：** 引入 NSQ 之前系统有什么问题？

**答：** 引入前主要是同步写库拖慢接口。[...1.2s]
我们把写事件改成异步投递到 NSQ，消费者落库。[...2.0s]
事务上靠消息 ID 加去重表保证幂等。

**问：** 吞吐提升 1.6 倍是怎么测出来的？关注哪些指标？

**答：** 对，是压测的……（未给出具体指标）

---
```

要点：
- `**问：**` / `**答：**` 各占一行，中间不插空行
- **长答换行**：步骤、分点、停顿标记后换行；避免一整段挤在一行
- 每组问答后空一行，再接 `---` 或下一组
- 候选人卡壳、忘记、答偏：在答末尾括号标注
- **分割/整理时尽量不删字**：只改明显错别字与 Whisper 误识别，勿为简洁而压缩原话

---

## iv{N}.md 模板

```markdown
[问题]
1. 背景 / 项目匹配
    1. 经验主要在 NAS，与 ADP 广告业务的关联
    2. B2C 还是 B2B；面向海外市场还是国内
2. NSQ 消息队列（云端项目）
    1. 引入 NSQ 之前系统有什么问题；为什么需要异步化
    2. 引入前链路里最慢的部分是什么
    3. 哪些操作做了异步化；写事件异步化带来的性能提升
    4. 事务与幂等怎么设计；会不会重复消费；消息 ID + 去重表
    5. Topic 怎么设计；消息队列整体架构 / Topic 拆分
    6. 吞吐提升 1.6 倍、延迟降低 50% 怎么测；压测指标是什么
3. 广告出价接口设计（场景题）
    1. 万级 QPS；100ms 内返回出价结果；怎么设计接口
    2. 日志要不要同步落库；更好的日志处理方式
    3. 超时控制怎么做（Context deadline）

NSQ、文件同步、限流缓存、广告出价、频控、MySQL
```

要点：
- 大类用 `N. 主题（可选项目/场景标注）`
- 子项用 4 空格缩进 + `N.`
- 最后一行标签：本轮涉及的技术/业务关键词
