## 广告出价接口设计

**背景**：服务收到万级 QPS；收到请求后需解析、过滤、打分、出价，**100ms 内**返回 Response；还要收集日志（什么流量、是否参与出价、打分多少）。

**主问**：
1. 如果是你，怎么设计这个接口？
2. 怎么确定一定是 100ms 以内返回？接口超时控制怎么做？
3. 日志是落库吗？你觉得日志要不要**同步**落库？除消息队列外还有什么更好的处理方式？
4. Goroutine 的生命周期在这个超时场景下怎么走？有没有遇到过 Goroutine 泄漏？

## 重复请求 / 防抖去重

**背景**：第一个请求发出后，抖动还没结束又发了第二个请求。

**主问**：如何做去重，保证不重复操作？

**要点方向**：
1. **客户端**：防抖（停止 n ms 再操作）、等响应或超时再恢复
2. **服务端**：唯一键 + sync.Map / Redis / 数据库，保证**最多执行一次**

## 家庭 AI 设备远程访问（系统设计）

**背景**：
- 家庭里有一台 AI 电脑：上电 → 用户**扫码绑定** → 任意时间地点**外网访问**设备上的服务（照片、本地 AI、设备状态等）

**主问**：
1. 请设计相关系统
2. 扫码绑定后，手机如何与设备绑定？异地如何远程访问？（P2P / Relay / Cloudflare Tunnel / 自建 Tunnel——怎么选、怎么落地）
3. 私密数据（照片、对话、账号密码）：如何让一部分可被本地模型学习、一部分不让学？云端应该看到什么、不应该看到什么？
4. 全球大量设备 **OTA** 推送更新，更新失败导致设备起不来，怎么办？

## 设备心跳（编码 / 设计题）

**背景**：简化版——设备每 **10 秒**向云端上报状态；云端判断设备是否在线；用户可随时查询设备在线/离线。

**主问**：
1. 写出核心结构体、接口、主要逻辑，以及你怎么测
2. 上报接口（如 POST）需要携带哪些字段？（device id、timestamp、meta、status 等）
3. 查询设备在线状态的接口怎么设计？
4. 心跳每 10 秒一次，若 **30 秒内**没有收到心跳，如何判定？
5. 设备断电时，如何知道它是 offline？
6. 有人伪造 device id 持续伪造心跳，怎么防？
