# 一、穿透
## 客户端、服务端、云端及穿透端的通信流程

**情境** 

1. 核心问题是 NAT 会拦截无状态入站流量：内网设备只有先出站建立映射，外部回包才进得来
2. 所以要进行双方通信，会通过P2P穿透，那要进行穿透就得知道对方的公网地址
3. 而我负责的项目主要是信令服务器，在双方还没建立起直连通道之前，提供一个互相交换信息的媒介

**角色**

- 客户端：手机
- 服务端：nas
- 云端：我负责的项目，信令服务器
- 穿透端：公网上的"穿透服务器"

**链路**

1. 首先，都在同一局域网的时候，手机点击开启穿透，nas会请求云端领一些东西
    1. 穿透凭证，用于nas开启穿透
    2. 验证码，用于手机向云端获取信息，在明，可展示给用户
    3. 接头暗号，用于验证身份
2. 然后nas会拿着穿透凭证，去公网上的"穿透服务器"那里注册或创建会话
3. 手机现在在外面了，把验证码给云端，问他这人在哪，云端查到后就给他穿透测入口，并给他接头暗号
4. 最后就是完成身份校验与通道建立

## 穿透用到什么技术，如何通信

1. 用户层，我们是通过http over tcp访问本机 127.0.0.1:50080，穿透映射远程的50080端口到本机
2. 通道层：底层通道由穿透网络提供，优先使用 P2P 直连；直连失败时自动切到节点转发 / 中继转发
3. 内部实现上（由第三方 SDK 封装）会做：NAT 类型探测、 STUN、打洞什么的

## 怎么得到的公网 IP 地址和端口？

1. 在原理中，是通过stun服务器获取的，设备向stun服务器注册内网地址，服务器会记录公网地址
2. 在我们的实现中，客户端不需要得到公网地址，只需要知道对端是谁，然后调用穿透sdk封装了ConnectAdd即可

## 为什么要引入消息队列

1. 解耦。让HTTP只负责业务逻辑，写事件交给消息队列处理。
2. 削峰。比如固件升级后大量设备重启、某个时间段用户集中远程访问 NAS。NSQ 可以让 HTTP 请求快速返回，然后消费者按数据库能承受的速度慢慢写。

## 为什么使用 NSQ

我选 NSQ 是因为这个项目的消息场景比较简单，主要是把操作记录、连接记录从 HTTP 主链路里异步出去，起到削峰和解耦作用。
它不需要 Kafka 那种大规模日志流能力，也不需要 RabbitMQ 很复杂的路由模型。
NSQ 部署轻量，Go 客户端成熟，topic/channel 模型简单，支持失败重试和队列积压观测，和 Go 服务集成成本低，所以更适合这个项目的规模。

### 为什么不用channel

只适合单进程内异步，不能跨实例，重启丢消息，积压不可观测

## 为什么要记录事件

1. **排障与还原现场**：用户反馈「连不上 / 凭证无效」时，可按时间线对照事件，避免只靠口述猜。
2. **安全与审计**：凭证颁发、身份校验、敏感操作留痕，满足**责任追溯**；出现异常访问或滥用时，有据可查。
3. **运营与产品分析**：统计成功率、失败原因分布、地域或版本维度的问题占比，用于优化 SDK 策略或容量规划。

# 三、加密
## 为什么要加密

若在 VPN 场景，常见链路是：`设备 -> [VPN 隧道] -> VPN 节点 -> [公网 TLS] -> Cloudflare`。

有些公司或特殊网络会要求在设备上安装其自有根证书（Root CA）。此时需注意：

- 一旦你信任了该根证书，VPN/代理节点理论上可以做“解密后再加密”（TLS 卸载 / 中间人代理）。
- 发生了两次https请求

## 双方加密流程

角色是Host设备和云端通信

1. Host设备rand生成AES密钥，使用AES-GCM加密明文，得到data,nonce
2. Host设备使用**云端**的RSA公钥加密AES密钥，得到key
3. Host设备将data,nonce,key发送给云端

4. 云端调用DecryptAESString，使用**云端**RSA私钥解密key，得到AES密钥
5. 云端使用AES密钥解密data,nonce，得到明文
6. 云端调用EncryptAESString，再生成新AES，使用**Host**的RSA公钥加密返回

7. Host端用**自己**的RSA私钥解密AES密钥，再解密响应
