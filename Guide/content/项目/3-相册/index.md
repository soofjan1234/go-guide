# 一、算法
## RKNN 平台是什么？和你的nas有什么关系

推出基于瑞芯微芯片（如RK3568、RK3588）的 ARM 架构 NAS，可以支持 RKNN 平台，通过 JNI 接口调用 native 层能力，实现模型推理。

降本增效：相比于 Intel 的 x86 芯片，价格更便宜、功耗极低，因为有 NPU 的存在，它的 AI 算力能直接吊打几千元的 Intel 处理器。

## 人脸检测和识别的流程

1. 人脸检测和识别：使用了 OpenCV ，返回人脸的位置信息并提取人脸特征向量
2. 人脸聚类：计算人脸特征之间的距离矩阵，再使用DBSCAN聚类，并得到每个类别的中心特征向量
3. 人脸分类：计算新人脸特征与聚类中心特征的距离，根据阈值来确定是加入还是创建新类别

## DBSCAN算法

拿社交圈举例：

1. 两个门槛：
    1. **eps（邻域半径）**：**"社交距离"**。就是你觉得多近才算"朋友圈"。比如 1 米内。
    2. **MinPts（最小点数）**：**"成群结队门槛"**。一个人带几个小弟才算"社交达人"？比如 3个。
2. 三个身份
    1. **核心点（大佬）**：1米内有3个
    2. **边界点（小弟）**：1米内不足3个，但在大佬的1米内
    3. **噪声点（独行侠）**：身边没人，也不在大佬圈子里
3. 聚类流程
    1. 随机抓一个人，看是否是大佬，是就把小弟都拉进来
    2. 再看小弟，是否也是大佬，如此套娃
    3. 知道所有人都被判定身份

## DBSCAN的技术选型

人脸聚类有K-Means、DBSCAN和HDBSCAN。
1. **K-Means**：简单、易于理解；需要预先指定簇的数量。
2. **DBSCAN**：适合处理噪声和复杂的簇形状；不需要预先指定簇的数量。
3. **HDBSCAN**：更智能的聚类，处理复杂分布更稳健；比 DBSCAN 慢一些。

## 相册 AI 用了什么模型？走了CPU吗

只有「场景/标签分类」走了 NPU（RKNN）；「人脸」没有走 NPU。

## 新照片来了怎么分类？

1. 不重新聚类，而是计算新人脸与所有**聚类中心**的余弦距离
2. 距离 < 阈值 → 归入已有聚类
3. 距离 > 阈值 → 创建新聚类，该特征向量作为新聚类中心

## 照片的选取测试有什么讲究

人脸检测用了WIDER val 全量 3226 张
人脸识别/验证用了 LFW (Labeled Faces in the Wild) 人脸识别数据集 1000 pairs，重点是“同一人 / 不同人”判断

## RKNN + JNI 这块你具体做了什么？

RKNN 主要用于端侧模型推理。流程上先把训练好的模型转换成 RKNN 支持的格式，然后在 native 层加载模型、初始化 RKNN runtime，并封装输入预处理、推理、输出解析这些步骤。
JNI 这块主要是把 native 层能力暴露给 Android 层，比如：
initModel()
classifyImage()
extractFaceFeature()
release()
Android 层负责图片读取、任务调度和结果入库，native 层负责模型推理。为了减少 JNI 调用开销，会尽量批量处理图片，并避免频繁传递大对象，图片会先做 resize 和格式转换，只传递推理所需的数据。


# 二、模糊、重复检测

整体流程：**先模糊检测 → 只对清晰照片做重复检测**（模糊图不参与 dHash 比对，减少无效计算）。
---

## 模糊算法

**拉普拉斯方差（Laplacian Variance）**

拉普拉斯算子用来看**边缘/细节**：细节多 → 方差大 → 越清晰；细节少（糊）→ 方差小 → 越模糊。

**处理流程**（`util/blur/blur.go`）：

1. 解码图片（支持 JPEG / PNG / GIF / WebP）
2. **缩放到固定宽度 512px**（保持宽高比，双线性插值），降低大图计算量
3. 转灰度图，再求**方差**作为清晰度分数 `score`
5. `score < blur_threshold` → 判定为模糊，写入 `blur_photos` 表

## 重复照片

- **dHash（差分哈希，Difference Hash）** + **pHash（感知哈希）**，双哈希联合判定。
- 比较方式：**汉明距离（Hamming distance）**，范围 0～64；距离越小越相似。

**三关匹配**（`PairMatch`，缺一不可）：

1. **宽高比**：`|宽/高_a - 宽/高_b| ≤ 0.05`（5%），过滤不同裁剪比例
2. **dHash 汉明距离** `≤ duplicate_threshold`（默认 4）
3. **pHash 汉明距离** `≤ duplicate_threshold`（默认 4）

**汉明距离参考**：

| 距离 | 含义 |
|---|---|
| 0 | 完全相同（复制文件） |
| 1～4 | 几乎相同（轻微裁剪/缩放/调色）→ **默认判重复** |
| 5～10 | 相似（连拍、编辑版） |
| 16+ | 不同照片 |

**流程**：

1. **先做模糊检测**：只对**非模糊**的清晰照片列表 `clearPhotos` 做重复检测。
2. **单次解码**同时算 dHash + pHash（`ComputeDPHashesFromImage`），结果可写入 **sha_hash 缓存**，避免重复算哈希。
3. **两两比较**：按清晰度分数**降序**排序，最清晰的作 **Keeper**（保留图），其余相似的归入同组 `Duplicate`。
4. 全量模式：对用户全部清晰图做检测；**增量模式**：新图与全库已有哈希比对，命中则并入旧组或新建组。

**Keeper 选择优先级**：清晰度最高 → 分辨率更大 → 拍摄时间更早（用户也可手动切换 Keeper）。

## 为什么选这两个

| 拉普拉斯方差（模糊） | dHash + pHash（重复） |
|---|---|---|
| **选型理由** | 纯 Go、无模型依赖，NAS 端侧 CPU 即可跑；比深度学习轻量，万级照片可接受 | 指纹固定 64 bit，算得快、占内存小；对缩放/轻微裁剪/调色不敏感 |
| **不选别的** | 不用 NPU/GPU（人脸和场景分类才走 RKNN）；梯度/FFT 实现更重或调参成本高 | 不用像素级逐字节比对（太慢）；单 dHash 误报多，加 pHash + 宽高比三关降误报 |


# 三、优化

## 通过增量扫描、特征缓存和批量处理

因为模糊、重复检测比较耗时，所以做了三层优化，对应简历里「增量扫描、特征缓存、批量处理」。

---

### 1. 增量扫描

默认模式 **`incremental`**，只处理「上次已完成任务」之后新增或修改的照片；**`full`** 才清空旧结果全量重跑。

**怎么界定增量范围**：

- 取最近一次 **completed** 任务的 `start_time` 作为基准
- 查 `media_infos` 中 `created_at > since` **或** `modified_at > since` 的照片
- 无历史任务 → 退化为全量


### 2. 特征缓存

重复检测最贵的是**解码 + 算 dHash/pHash**；同一张图（SHA1 不变）不应每次重算。

**`sha_hash_caches` 表**：`SHA1 → dHash + pHash`，跨任务、跨扫描复用。

**`hydratePhotoHashes` 流程**：

1. 收集本批清晰图的 SHA1 列表
2. **`GetShaHashCachesBySHA1s` 批量 IN 查询**缓存
3. 命中 → 直接反序列化 hash，仅缺 pHash/宽高时才补算
4. 未命中 → `LoadPhotoHashInfo` 解码一次，同时算 dHash + pHash
5. **`UpsertShaHashCaches` 批量回填**缓存

**模糊分数也复用**：模糊检测阶段算出的 `ClearScore` 写入 `clearPhotos`，重复检测选 Keeper 时不再重算拉普拉斯。

增量重复检测还会从 `GetUserLibraryDHashRows`（媒体表 ⋈ sha 缓存）拉全库参考集，避免对老图重新解码。

### 3. 批量 / 并发处理

**模糊检测**（`detectBlur`）：

- Worker 数 = `min(NumCPU, 8, 照片数)`，并行算拉普拉斯方差
- **写库顺序执行**（适配 SQLite 单连接写，避免锁竞争）
- 清晰图结果带 `SHA1`、`ClearScore` 直接传给下一阶段

## 还有什么优化空间？

这个重复检测可以做算法的优化：

1. hash 分桶 / 前缀过滤：dHash 高几位相同才进同一桶，桶内再比汉明距离
2. BK-tree / VP-tree：在汉明距离度量空间里做近邻搜索

全量重复检测理论上是 O(n²)，但单次 PairMatch 只是 64 bit 汉明距离，真正贵的是图片解码，所以我们用 SHA1 特征缓存和增量扫描把 n² 限制在「首次全量」场景。

万级清晰图以内纯比对通常还能接受；若用户库到两万、三万，可以用 hash 分桶或 BK-tree 做近邻搜索，或并行化两两比对。是否优化要看监控数据，不能为了 O(n log n) 过早引入 LSH 带来漏检风险。
