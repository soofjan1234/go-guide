# 一、算法
## RKNN 平台是什么？和你的nas有什么关系

推出基于瑞芯微芯片（如RK3568、RK3588）的 ARM 架构 NAS，可以支持 RKNN 平台，通过 JNI 接口调用 native 层能力，实现模型推理。

降本增效：相比于 Intel 的 x86 芯片，价格更便宜、功耗极低，因为有 NPU 的存在，它的 AI 算力能直接吊打几千元的 Intel 处理器。

## 人脸检测和识别的流程

1. 人脸检测和识别：使用了 OpenCV ，返回人脸的位置信息并提取人脸特征向量
2. 人脸聚类：计算人脸特征之间的距离矩阵，再使用DBSCAN聚类，并得到每个类别的中心特征向量
3. 人脸分类：计算新人脸特征与聚类中心特征的距离，根据阈值来确定是加入还是创建新类别

## DBSCAN算法

拿社交圈举例：

1. 两个门槛：
    1. **eps（邻域半径）**：**"社交距离"**。就是你觉得多近才算"朋友圈"。比如 1 米内。
    2. **MinPts（最小点数）**：**"成群结队门槛"**。一个人带几个小弟才算"社交达人"？比如 3个。
2. 三个身份
    1. **核心点（大佬）**：1米内有3个
    2. **边界点（小弟）**：1米内不足3个，但在大佬的1米内
    3. **噪声点（独行侠）**：身边没人，也不在大佬圈子里
3. 聚类流程
    1. 随机抓一个人，看是否是大佬，是就把小弟都拉进来
    2. 再看小弟，是否也是大佬，如此套娃
    3. 知道所有人都被判定身份

## DBSCAN的技术选型

人脸聚类有K-Means、DBSCAN和HDBSCAN。
1. **K-Means**：简单、易于理解；需要预先指定簇的数量。
2. **DBSCAN**：适合处理噪声和复杂的簇形状；不需要预先指定簇的数量。
3. **HDBSCAN**：更智能的聚类，处理复杂分布更稳健；比 DBSCAN 慢一些。

只有「场景/标签分类」走了 NPU（RKNN）；「人脸」没有走 NPU。

## RKNN + JNI 这块你具体做了什么？

RKNN 主要用于端侧模型推理。流程上先把训练好的模型转换成 RKNN 支持的格式，然后在 native 层加载模型、初始化 RKNN runtime，并封装输入预处理、推理、输出解析这些步骤。

JNI 这块主要是把 native 层能力暴露给 Android 层，Android 层负责图片读取、任务调度和结果入库，native 层负责模型推理。为了减少 JNI 调用开销，会尽量批量处理图片，并避免频繁传递大对象，图片会先做 resize 和格式转换，只传递推理所需的数据。


# 二、模糊、相似检测
## 模糊算法

选拉普拉斯方差做模糊检测，是因为模糊图片的边缘和高频细节会明显减少，而拉普拉斯算子正好对边缘变化敏感。

清晰图的拉普拉斯方差通常更大，模糊图更小，所以可以用一个阈值快速判断清晰度。

### 其他方法

除此之外还可以使用 Sobel 等梯度方法，还可以训练 CNN 模型直接输出图像清晰度评分。

对于工程实践，Laplacian Variance 因为简单高效，是最常见的第一层筛选方案。

## 相似照片

第一类是感知哈希算法，例如 aHash、dHash、pHash，通过生成视觉特征哈希并计算汉明距离判断相似度；
- pHash 是感知哈希，它更关注图片整体视觉特征
- dHash 是差异哈希，它更关注图片局部特征

第二类是局部特征匹配，例如 SIFT、ORB，通过关键点和描述子匹配解决裁剪、旋转和尺度变化问题；

第三类是深度学习方法，使用 ResNet 等模型提取图片 Embedding，再通过余弦相似度或向量检索判断相似度。

# 三、优化

## 通过增量扫描、特征缓存和批量处理

因为模糊、重复检测比较耗时，所以做了三层优化，对应简历里「增量扫描、特征缓存、批量处理」。

1. 增量扫描

- 取最近一次完成的任务的开始时间作为基准
- 查照片中 创造时间或修改时间大于基准时间的照片

2. 特征缓存

- `SHA1 → dHash + pHash`，跨任务、跨扫描复用。
- 模糊检测阶段算出的 `ClearScore` 写入 `clearPhotos`，重复检测选 Keeper 时不再重算拉普拉斯。

3. 批量 / 并发处理

- Worker 数 = `min(NumCPU, 8, 照片数)`，并行算拉普拉斯方差
- **写库顺序执行**（适配 SQLite 单连接写，避免锁竞争）
- 清晰图结果带 `SHA1`、`ClearScore` 直接传给下一阶段

## 还有什么优化空间？

这个重复检测可以做算法的优化：

1. hash 分桶 / 前缀过滤：dHash 高几位相同才进同一桶，桶内再比汉明距离
2. BK-tree / VP-tree：在汉明距离度量空间里做近邻搜索

全量重复检测理论上是 O(n²)，但单次 PairMatch 只是 64 bit 汉明距离，真正贵的是图片解码，所以我们用 SHA1 特征缓存和增量扫描把 n² 限制在「首次全量」场景。

万级清晰图以内纯比对通常还能接受；若用户库到两万、三万，可以用 hash 分桶或 BK-tree 做近邻搜索，或并行化两两比对。是否优化要看监控数据，不能为了 O(n log n) 过早引入 LSH 带来漏检风险。
