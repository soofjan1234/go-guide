## 2. 云端 / NAT 穿透

### 表现

这一段内容准备得比较多，但讲得偏绕。

你讲到了：

1. NAT 不能接受陌生入站流量。
2. 客户端、服务端、云端、穿透端四个角色。
3. 云端做信令交换。
4. 邀请码、校验码、穿透凭证。
5. 对称 NAT 下端口变化，打洞容易失败。
6. 失败后走转发服务。

这些点是对的。

### 扣分点

1. “云端在同一局域网下”这句话容易误导。
2. “服务端、穿透端、客户端”角色有时混用。
3. 对称 NAT 的解释停顿很多，端口例子没有一口气讲清楚。
4. 转发服务的解释有点抽象，可以直接说 Relay。

### 建议回答

```text
这个链路里主要有四个角色：
手机 App、NAS 服务端、云端信令服务、穿透 / Relay 服务。

首先，因为 NAS 通常在家庭内网后面，外网不能直接访问它，所以需要 NAT 穿透。
云端不直接转发业务数据，主要负责信令交换，比如交换双方的公网地址、端口、连接凭证和校验信息。

正常情况下，App 和 NAS 都先向云端注册自己的连接信息。
当用户开启远程访问时，App 通过云端拿到 NAS 的连接信息，然后双方尝试 P2P 打洞。

如果 NAT 类型比较友好，比如全锥型、受限锥型，P2P 成功率会高。
但如果是对称型 NAT，同一个内网地址访问不同目标时，映射出来的公网端口会变化。
这会导致云端交换到的地址端口，和真正连接时使用的地址端口不一致，所以直接打洞容易失败。

失败时我们会降级到 Relay 转发，也就是 App 和 NAS 都连接到中转服务，由中转服务转发流量。
这样延迟和带宽成本会更高，但能保证可用性。
```

## 3. TLS 之外为什么还要加密

### 表现

你答到了一个关键点：公司或 VPN 可能安装根证书，中间代理能解 TLS，所以再做一层业务加密。

### 扣分点

密钥分发回答很弱，说“企业微信直接有”，这会非常扣分。安全问题不能回答得像手动发密钥。

### 建议回答

```text
TLS 解决的是传输层加密，但在一些企业网络或代理环境里，如果用户安装了企业根证书，中间代理可能对 HTTPS 做解密和审计。

所以我们在业务层又做了一层加密，目的是即使 TLS 被中间代理终止，业务敏感字段仍然不能被直接读取。

一般做法是：使用非对称加密交换临时密钥，再用对称加密保护业务数据。
比如客户端拿到服务端公钥，用公钥加密一个随机生成的 AES key，服务端用私钥解密后，后续双方用 AES 做数据加密。

这样既保证密钥传输安全，也避免 RSA 直接加密大数据带来的性能问题。
```

## 5. 文件索引同步器

### 表现

这段明显卡。你知道 `fsnotify` 的问题，但没有说成系统方案。

你讲到了：

1. fsnotify 是文件监听。
2. 每个文件夹 watch 会占资源。
3. 文件多时可能漏事件。
4. 后面改成队列 + 遍历同步。
5. 修改时间和大小不变则跳过索引。

### 扣分点

1. BFS / DFS 说法混乱。
2. 百万级遍历耗时没数据，容易被追问。
3. 500 万文件系统索引是开放题，你没有展开方案。
4. 面试官提示 NTFS MFT，说明他期待你能从“文件系统元数据”角度思考。

### 建议回答

```text
fsnotify 适合做增量监听，但它有两个问题。

第一，监听目录越多，watch 资源和内存开销越大。
第二，当文件事件特别密集，或者目录层级很多时，可能出现事件丢失或处理不及时。

所以我们后面不是完全依赖 fsnotify，而是做了“监听 + 定期扫描”的兜底。
监听负责捕获实时变化，定期扫描负责发现漏掉的文件。

扫描时我会用队列控制并发，不会一次性把所有路径放进内存。
每个文件会对比 mtime、size 等元数据，如果没有变化，就跳过重新索引。
这样能减少大量无效 IO 和重复索引。

如果是 500 万文件量级，我会分几层设计：
第一，扫描任务分批，控制并发和内存；
第二，保存文件元数据表，用 mtime、size、inode 或文件 ID 判断是否变化；
第三，索引构建异步化，扫描和建索引解耦；
第四，如果是 Windows NTFS，可以考虑利用 MFT 或系统索引接口，减少全盘遍历成本。
```

## 6. GMP

### 表现

你准备了很多字段，但答得像背源码字段，容易显得乱。

### 建议回答

```text
Go 的 GMP 模型里：

G 是 Goroutine，代表一个用户态执行任务，里面有栈、状态、待执行函数等信息。

M 是 Machine，可以理解为 OS 线程，真正执行代码的是 M。

P 是 Processor，可以理解为调度上下文。P 里有本地 runnable 队列、runnext，还有一些缓存。
M 必须绑定 P，才能执行 G。

调度时，P 会优先从本地队列取 G 执行；本地没有时，会从全局队列拿，或者从其他 P 的队列里 work stealing。

P 的数量由 GOMAXPROCS 决定，默认通常等于 CPU 核数。
```

## 7. NSQ

### 表现

方向正确，但术语要说准：`NSQ`、`nsqd`、`Topic`、`Channel`、`ACK`、`持久化`。

### 建议回答

```text
NSQ 是一个去中心化的消息队列，部署简单，比较适合中小规模异步任务。

核心组件是 nsqd、nsqlookupd 和 nsqadmin。
nsqd 负责真正接收、存储和投递消息。
生产者把消息发到 Topic，Topic 会把消息复制到不同 Channel。
同一个 Channel 下多个消费者是竞争消费，用来做负载均衡。

相比 Kafka，Kafka 更适合高吞吐日志和事件流平台，依赖 broker 集群和分区。
NSQ 更轻量，适合我们这种业务量没那么大、希望部署简单的场景。

消息丢失主要有几种情况：
第一，生产者发送失败；
第二，nsqd 没有开启持久化或还没落盘就宕机；
第三，消费者处理时机和 ACK 不合理。

解决方式是生产端检查发送结果并重试，nsqd 开启持久化，消费者处理成功后再 ACK，失败则 requeue。
重复消费则通过业务唯一 ID 做幂等。
```

## 10. 编程题 / MQTT 订阅匹配

### 表现

这段是本轮最大扣分点。

停顿特别长，说明你对题目理解不稳。后来在面试官提示下，你能说到 `pattern -> handler` 和 `publish(topic)` 去匹配 pattern，但这应该更早说出来。

### 正确思路

```text
这个题可以抽象成一个订阅和发布系统。

subscribe(pattern, handler) 的时候，把 pattern 和对应 handler 保存起来。
最简单的实现是一个 map 或数组：
pattern -> handlers。

publish(topic, message) 的时候，遍历所有 pattern，判断 topic 是否匹配。
如果匹配，就调用对应 handler，把 message 传进去。

如果支持 MQTT 的 `+` 通配符，可以把 pattern 和 topic 都按 `/` 分割成数组。
然后逐段比较：
如果 pattern 当前段是 `+`，就匹配任意一段；
否则必须和 topic 当前段相等；
最后长度也要一致。

如果要优化性能，可以用 Trie，把 topic 的每一段作为树的一层，订阅时把 pattern 插入 Trie，发布时沿着 topic 逐层匹配，同时处理 `+` 分支。
```

### Go 伪代码

```go
type Handler func(Message)

type Broker struct {
    subs map[string][]Handler
}

func (b *Broker) Subscribe(pattern string, h Handler) {
    b.subs[pattern] = append(b.subs[pattern], h)
}

func (b *Broker) Publish(topic string, msg Message) {
    for pattern, handlers := range b.subs {
        if match(pattern, topic) {
            for _, h := range handlers {
                h(msg)
            }
        }
    }
}

func match(pattern, topic string) bool {
    ps := strings.Split(pattern, "/")
    ts := strings.Split(topic, "/")
    if len(ps) != len(ts) {
        return false
    }
    for i := range ps {
        if ps[i] == "+" {
            continue
        }
        if ps[i] != ts[i] {
            return false
        }
    }
    return true
}
```
