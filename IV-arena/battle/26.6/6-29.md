1. 线上事故排查回答需要更工程化

当前回答偏向“登录机器看 CPU/内存/pprof”，容易被追问监控、日志、链路追踪时卡住。建议固定成一套排查框架：

先确认影响面：失败用户数、失败比例、错误码、地域、版本、时间窗口、是否集中在某个入口。

再看变更：最近发布、配置变更、依赖服务变更、数据库/缓存/网关变更。

然后看观测数据：接口 QPS、错误率、P95/P99、Nginx/网关日志、业务日志 traceId、数据库慢查询、Redis/DB 连接池、第三方依赖超时。

最后做止血：回滚、降级、限流、扩容、关闭新功能开关、补偿失败订单。

2. Go channel 防阻塞是明显短板

面试官已经明确提示：不能靠“缓冲区足够大”。标准答案要落到：

```go
select {
case ch <- v:
    // sent
case <-time.After(200 * time.Millisecond):
    // timeout, drop / fallback / return error
case <-ctx.Done():
    return ctx.Err()
}
```

接收也同理。关键词是 `select`、`time.After` / `timer`、`ctx.Done()`、超时后降级或返回。

3. Goroutine 关闭要说清“协作式退出”

Go 不能从外部强杀 Goroutine。常见做法是：

用 `context.WithCancel/WithTimeout` 传递取消信号；
Goroutine 内部在循环或阻塞点监听 `ctx.Done()`；
配合 channel 关闭、select、timer 做超时兜底；
函数内部收到信号后主动释放资源并 return。

4. slice 题需要补上扩容条件

slice 传参复制的是 slice header：底层数组指针、len、cap。修改已有元素会影响共享同一底层数组的 slice；append 是否影响原 slice 取决于是否扩容。

未扩容时，新元素写入同一底层数组，可能影响其他 slice 能看到的底层数据；扩容后会分配新数组，后续修改不再影响原底层数组。

5. MySQL explain 要背熟 possible_keys / key

`possible_keys` 是优化器认为可能可用的索引集合，`key` 是最终执行计划实际选择的索引。实际索引和预期不同，常见原因包括：

字段类型不一致导致隐式转换、对索引列做函数/计算、最左前缀不满足、选择度太低、统计信息不准、优化器认为全表扫或另一个索引成本更低。

6. Redis 持久化回答需要修正

Redis 同时开启 RDB 和 AOF 时，通常优先使用 AOF 恢复，因为 AOF 数据更完整；如果启用 AOF 混合持久化，AOF 文件前半部分是 RDB 格式快照，后半部分是增量 AOF 命令。

过期删除可以说：惰性删除 + 定期删除；内存不足时再配合内存淘汰策略，不要把随机抽查说成唯一机制。

7. Kafka 分区消费者题

同一个 consumer group 内，一个 partition 同一时刻只能分配给一个 consumer。5 个 partition、6 个 consumer 时，会有 1 个 consumer 空闲，不会参与消费；不会因为消费者更多就自动提高并行度。

8. TCP TIME_WAIT 题

TIME_WAIT 的核心目的：确保最后一个 ACK 能被对端收到，以及让旧连接的延迟报文在网络中自然消失，避免污染后续相同四元组连接。

大量 TIME_WAIT 的处理方向：连接复用、HTTP keep-alive、调整短连接模型、扩容端口/实例、合理内核参数。不要回答“超时就直接关闭资源”。
