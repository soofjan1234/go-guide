[答案]
1. sync.Map 的使用优化

- 首先是 场景要对：一写多读、或各 goroutine 访问不同 key；不要用来扛频繁删改、多 goroutine 抢同一 key；
- 用法上多用 LoadOrStore、LoadAndDelete、CAS 系 API，热路径 少 Range、少 Delete；大 value 用指针
- 不满足这些就用 map + 锁 或 分段锁，别为了「并发安全」盲目换 sync.Map。

2. 分段锁

- 把大 map 按 hash 分成多段，每段一个小 map 配一把锁，访问 key 只锁对应段，减少竞争。
- Go 标准库没有内置，高并发、读写都多、又要频繁更新时，常自己实现分段 map，或用 map+RWMutex
- 若是一写多读、key 稳定，才更偏向 sync.Map

3. 一万个人的数据存string还是hash，为什么

- 一万个用户、每人多字段：按用户拆 key，用 Hash（user:{id} + field）；单字段用 String；别用一个 Hash 装所有人。
- 选 Hash 是因为内存更省、支持字段级读写、避免 JSON 整读整写和并发覆盖。
