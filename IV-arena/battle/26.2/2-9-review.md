## 1. 云端通信 / P2P 穿透

### 1.1 请讲一下云端通信的整体流程

**情境**

1. 核心问题是 NAT 会拦截无状态入站流量：内网设备只有先出站建立映射，外部回包才进得来
2. 所以要进行双方通信，会通过 P2P 穿透，那要进行穿透就得知道对方的公网地址
3. 而我负责的项目主要是信令服务器，在双方还没建立起直连通道之前，提供一个互相交换信息的媒介

**角色**

- 客户端：手机
- 服务端：NAS
- 云端：我负责的项目，信令服务器
- 穿透端：公网上的「穿透服务器」

**链路**

1. 首先，都在同一局域网的时候，手机点击开启穿透，NAS 会请求云端领一些东西
    1. 穿透凭证，用于 NAS 开启穿透
    2. 验证码，用于手机向云端获取信息，在明，可展示给用户
    3. 接头暗号，用于验证身份
2. 然后 NAS 会拿着穿透凭证，去公网上的「穿透服务器」那里注册或创建会话
3. 手机现在在外面了，把验证码给云端，问他这人在哪，云端查到后就给他穿透侧入口，并给他接头暗号
4. 最后就是完成身份校验与通道建立

### 1.2 外网为什么访问不到内网设备？NAT 映射表和入站流量拦截的关系是什么？

NAT 的核心原理是让局域网（私网）内的多台设备共享少量公网 IP。

**出站规则（NAPT 端口复用）**

- 私网主机 `192.168.1.100:12345` 访问外网时，NAT 路由器会把源 IP/端口替换为 `公网IP:6789`
- 同时在 **NAT 映射表** 中记录：`192.168.1.100:12345 ↔ 公网IP:6789`

**入站规则**

- 外网回包发到 `公网IP:6789` 时，NAT 查映射表，反向转发给 `192.168.1.100:12345`

**为什么外网主动访问进不来**

- 隐藏在 NAT 后面的主机，**只有它先对外发起连接、映射表里有记录时，外部回包才能进来**
- 如果外部网络想**主动**联系 NAT 内主机，映射表中没有对应条目，入站流量会被拒绝 → 这就是 P2P 穿透要解决的问题

### 1.3 如何得知对方的公网 IP？
**我们实现里的简化**

- 原理上通过 STUN 服务器获取公网地址
- 业务上客户端不需要自己拿公网地址，只需知道对端是谁，调用穿透 SDK 封装的 `ConnectAdd` 即可

## 2. Redis

### 2.1 缓存穿透、缓存击穿、缓存雪崩分别是什么？各自用什么典型场景解释？

**缓存穿透（查不存在的数据）**

- 场景：黑客用大量**不存在的商品 ID** 访问，请求既不在缓存也不在 DB，每次都打到数据库
- 方案：缓存空值、接口限流、布隆过滤器

**缓存击穿（热点 key 过期）**

- 场景：iPhone 16 是热点商品，缓存刚好过期，大量并发同时查 DB
- 方案：热点永不过期、过期前预热、分布式锁（只让一个请求回源）

**缓存雪崩（大面积同时失效 / Redis 不可用）**

- 场景：10 万人同时点 10 万个产品，缓存**同时过期**，或 Redis 集群整体不可用，大面积瘫痪
- 方案：Redis 集群 + 多级缓存；随机 TTL、提前预热、分批设置过期

### 2.2 布隆过滤器怎么理解？新数据写入和查询流程是什么？

布隆过滤器 = 超长位数组（Bitmap）+ 多个哈希函数，快速判断 key **是否可能存在**。

**核心结论**

- 说「不存在」→ 一定不存在
- 说「存在」→ 只是可能存在（有误判）

**写入流程**

1. 拿一个数据（如 `User123`）
2. 用多个哈希函数计算多个下标（如 10、50、80）
3. 把位数组对应位置都置为 1

**查询流程**

1. 对待查询 key 计算同样的多个下标
2. 只要有一个位置是 0 → 一定不存在
3. 全部是 1 → 可能存在（再查缓存/DB 确认）

**落地闭环**

- 新服务 → 先全量预热主键进布隆，或布隆 ready 前不过滤
- 线上创建成功就 Add
- 定期全量重建，或用计数布隆等结构处理删除（标准布隆不支持删）

### 2.3 Redis 持久化机制有哪些？RDB 和 AOF 的区别是什么？

**RDB（快照）**

- 将内存数据以二进制快照写到磁盘（`BGSAVE` + fork + 写时复制 COW）
- 触发：手动 `BGSAVE`、或 `save <seconds> <changes>` 自动触发
- 崩溃时：磁盘上要么还是上一份完整 RDB，要么本次临时文件作废

**AOF（追加日志）**

- 按时间顺序记录写命令到日志
- 流程：命令追加到 AOF 缓冲区 → write 到 Page Cache → fsync 落盘
- `appendfsync`：`always`（最安全）/ `everysec`（推荐，最多丢 1 秒）/ `no`
- AOF 重写：子进程按当前内存状态写新文件，主进程用 Rewrite Buffer 追增量，最后原子替换

**区别**

| 维度 | RDB | AOF |
| --- | --- | --- |
| 数据形式 | 某一时刻全量快照 | 写命令日志 |
| 丢失窗口 | 可能丢失上次快照后全部数据 | 通常更少（everysec 约 1 秒） |
| 文件/恢复 | 更小、恢复更快 | 更大、恢复较慢 |
| 生产建议 | 可接受丢数据时偏向 RDB | 更重视完整性时偏向 AOF；常见 RDB + AOF 一起开 |

### 2.4 Redis 的过期策略有哪些？惰性删除和定期抽样删除分别怎么工作？

**惰性删除（Lazy Deletion）**

- 只有访问（GET/SET 等）这个 key 时，才检查是否过期；过期则删除并返回空
- 优点：对 CPU 友好，不主动扫全库
- 缺点：对内存不友好，大量过期 key 没人访问会一直占内存

**定期删除（Periodic Deletion）**

- 默认每秒 10 次（100ms 一次）抽样扫描
- 从设置了过期时间的 key 中随机抽 20 个 → 删已过期的 → 若过期比例 > 25% 则继续扫

### 2.5 Redis 内存接近上限时会发生什么？淘汰策略可能有哪些？

惰性 + 定期删除都没清掉足够内存、达到 `maxmemory` 时，触发 **内存淘汰策略（Eviction Policy）**：

- `noeviction`（默认）：拒绝写入，报错
- `allkeys-lru`：所有 key 中淘汰最近最少使用的（最常用）
- `volatile-lru`：只在设置了过期时间的 key 中 LRU 淘汰
- `allkeys-lfu`：淘汰使用频率最低的
- `allkeys-random`：随机删（不推荐）

### 2.6 一个 key 被删除后是否真的立即物理删除？key 删除后 AOF 和 RDB 分别会有什么动作？

**逻辑 vs 物理删除**

- **逻辑层面**：执行 `DEL` 后，Redis 立即从 dict 中删掉 key 索引，`EXISTS`/`GET` 再也找不到
- **物理层面**：
  - 小 key：内存立即标记为空闲，可复用
  - 大 key（如百万元素 List）：可能后台线程异步释放（`UNLINK` 或新版可配置异步 `DEL`）
- **注意**：Redis 释放了内存，OS 不一定立刻回收物理内存（内存碎片）

**AOF**

- `DEL`/`UNLINK` 属于写命令，会追加到 AOF 缓冲区，按 `appendfsync` 策略落盘
- AOF 重写时按**当前内存状态**生成新文件，已删 key 不会出现在重写结果里

**RDB**

- RDB 是某一时刻快照，删除后**当前内存已无该 key**
- 下次 `BGSAVE` 成功的快照中不会包含该 key；上一份 RDB 文件内容不变，直到被新快照覆盖

## 3. MySQL

### 3.1 唯一索引和普通索引在实现上有什么区别？

**共同点**

- 都是二级索引（B+ 树），都能加速查找

**普通索引**

- 仅承担查询加速
- 插入/更新时，若目标索引页不在内存，可借助 **Change Buffer** 先缓冲变更，等索引页读入内存或后台 merge 时再合并，减少随机 I/O

**唯一索引**

- 除加速查找外，还承担**唯一性约束**
- 每次插入必须做 **Unique Check**：即使索引页不在内存，也要去磁盘读原页验证是否重复 → 必然随机 I/O，**无法使用 Change Buffer 优化**
- EXPLAIN 上：主键/唯一索引等值查常见 `const`；普通索引等值查常见 `ref`

> 为什么 Change Buffer 必须是非唯一索引？因为唯一索引插入时必须读磁盘做唯一性检查，页都读出来了，缓冲层失去意义。

### 3.2 MySQL 内存高或 CPU 高怎么排查？

> 注：Article 里 MySQL 专项排查较少，通用排查见 `Article/项目/排查.md`，索引/SQL 方向见 `3.2-索引应用.md`

**CPU 高**

1. 先分清「真 CPU 忙」还是「假高负载」：Load 高但 CPU 低，多半是 IO 等待、锁等待
2. `top` 看是 mysqld 还是别的进程
3. 常见原因：慢 SQL、全表扫描、锁竞争、连接数过多、buffer pool 不够导致频繁读盘

**内存高**

1. `free -h` 看 available；Linux cache 高不一定有问题
2. MySQL 侧关注：`innodb_buffer_pool_size`、连接数 × 每连接内存、sort/join buffer、临时表
3. 配合慢查询日志、`EXPLAIN`、`SHOW PROCESSLIST` 看是否有大查询/锁等待

**数据库慢（通用）**

1. 慢查询日志，看是否持续出现
2. `EXPLAIN` 是否走索引、`key_len` 命中几列、`type` 是否 `ALL`
3. 是否有锁等待；索引失效、回表过多、Cardinality 不准（可 `ANALYZE TABLE`）

## 4. 技术选型会考虑什么

**选型关键点（泛问题）**

1. 要解决的核心问题是什么
2. 先看业务规模与约束（QPS、数据量、资源上限）
3. 在满足需求的前提下比**部署复杂度、长期维护成本**
4. 最后看特定需求（中文分词、嵌入式、消息不丢等）
